# 001. Управление задачами — бизнес-требования

## 1. Цель

Реализовать базовое управление задачами в приложении `TK.Tasks`, чтобы пользователь мог создавать, просматривать, изменять, выполнять и удалять задачи, а также работать с приоритетами, категориями, избранным и подзадачами.

Фича должна превратить текущий frontend с временным `localStorage` в полноценный сценарий с backend API и PostgreSQL-хранилищем.

## 2. Область изменений

Фича затрагивает:

- `todo-back` — API, доменная модель, usecase, repository, миграции, OpenAPI;
- `todo-front` — RTK Query слой, интеграция с backend, UI-состояния загрузки/ошибок;
- `specs` — бизнес- и техническое описание.

## 3. Пользовательские сценарии

### 3.1. Просмотр задач

Пользователь может открыть страницу задач и увидеть список своих задач.

Список должен поддерживать:

- просмотр всех задач;
- просмотр задач на сегодня;
- просмотр предстоящих задач;
- просмотр выполненных задач;
- просмотр важных задач;
- фильтр по категории;
- поиск по названию;
- сортировку по времени;
- сортировку по приоритету.

### 3.2. Создание задачи

Пользователь может создать новую задачу.

Минимальные поля при создании:

- название;
- категория;
- приоритет.

Система должна автоматически назначать:

- дату — сегодня;
- время — текущее время;
- статус `done = false`;
- `starred = false`;
- пустые заметки;
- пустой список подзадач.

### 3.3. Просмотр деталей задачи

Пользователь может открыть задачу и увидеть:

- название;
- категорию;
- дату;
- время;
- приоритет;
- заметку;
- статус выполнения;
- статус важности;
- список подзадач;
- прогресс по подзадачам.

### 3.4. Изменение задачи

Пользователь может изменить:

- название;
- категорию;
- приоритет;
- дату;
- время;
- заметку;
- статус выполнения;
- статус важности.

На первом этапе UI может поддерживать не все поля редактирования напрямую, но backend API должен быть готов принимать частичное обновление задачи.

### 3.5. Выполнение задачи

Пользователь может отметить задачу выполненной или вернуть её в работу.

Выполненная задача должна:

- оставаться в списке;
- визуально отображаться как завершённая;
- учитываться в статистике прогресса.

### 3.6. Важная задача

Пользователь может отметить задачу как важную или снять отметку.

Важные задачи должны быть доступны через фильтр `Важное`.

### 3.7. Удаление задачи

Пользователь может удалить задачу.

После удаления:

- задача исчезает из списка;
- прямой переход в детали удалённой задачи должен показывать состояние “задача не найдена”.

### 3.8. Подзадачи

Пользователь может:

- добавлять подзадачи к задаче;
- отмечать подзадачи выполненными/невыполненными.

Подзадачи должны влиять на прогресс задачи.

Удаление подзадачи можно отложить на следующий этап, если оно не требуется для базового сценария.

### 3.9. Календарь

Пользователь может открыть календарь и увидеть дни, в которых есть задачи.

Для дня с задачами должны отображаться маркеры.

При клике по дню с задачами пользователь может перейти к первой задаче этого дня.

## 4. Категории задач

Поддерживаемые категории:

| Код | Название |
| --- | --- |
| `work` | Работа |
| `personal` | Личное |
| `health` | Здоровье |
| `learning` | Обучение |
| `home` | Дом |

## 5. Приоритеты задач

Поддерживаемые приоритеты:

| Код | Название |
| --- | --- |
| `high` | Высокий |
| `med` | Средний |
| `low` | Низкий |

## 6. Статистика

На dashboard должны отображаться данные, рассчитанные на основе задач:

- количество активных задач;
- количество выполненных задач;
- задачи на сегодня;
- прогресс дня;
- общий прогресс;
- предстоящие задачи.

На первом этапе статистика может рассчитываться на frontend из списка задач. Backend endpoint для статистики можно добавить позже, если потребуется оптимизация.

## 7. Бизнес-правила

1. Название задачи обязательно.
2. Пустое или состоящее только из пробелов название недопустимо.
3. Категория должна быть одной из разрешённых.
4. Приоритет должен быть одним из разрешённых.
5. Дата задачи хранится как календарная дата.
6. Время задачи хранится в формате `HH:mm`.
7. Удаление задачи удаляет связанные подзадачи.
8. Подзадача не может существовать без задачи.
9. Завершение всех подзадач не обязано автоматически завершать задачу.
10. Завершение задачи не обязано автоматически завершать все подзадачи.

## 8. Нефункциональные требования

### 8.1. Производительность

Для текущего этапа достаточно корректной работы на небольшом объёме данных: десятки/сотни задач.

### 8.2. Надёжность

Backend должен возвращать понятные ошибки:

- `400` — некорректный запрос;
- `404` — задача или подзадача не найдена;
- `500` — внутренняя ошибка.

### 8.3. Расширяемость

Архитектура должна позволять позже добавить:

- пользователей;
- авторизацию;
- владение задачами пользователем;
- напоминания;
- повторяющиеся задачи;
- полноценную статистику;
- админский API.

### 8.4. Документация

OpenAPI документация должна быть обновлена и доступна через Swagger UI:

```txt
/docs
```

## 9. Acceptance criteria

Фича считается реализованной, если:

1. Backend содержит миграции таблиц задач и подзадач.
2. Backend реализует REST API для задач и подзадач.
3. OpenAPI описывает новые endpoints.
4. Frontend использует backend API через RTK Query, а не временный `localStorage` adapter.
5. Пользователь может создать задачу во frontend, и она сохраняется в PostgreSQL.
6. Пользователь может увидеть список задач после перезагрузки страницы.
7. Пользователь может открыть детали задачи.
8. Пользователь может отметить задачу выполненной/невыполненной.
9. Пользователь может отметить задачу важной/неважной.
10. Пользователь может изменить приоритет задачи.
11. Пользователь может удалить задачу.
12. Пользователь может добавить подзадачу.
13. Пользователь может отметить подзадачу выполненной/невыполненной.
14. Dashboard и календарь работают на данных из backend.
15. Backend tests проходят.
16. Frontend build проходит.
